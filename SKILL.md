# 方法级接口文档 · HTML 版（method-api-doc-html）

为**单个 Java 方法**生成一份**格式固定、面向调用方**的接口契约文档，**默认输出为精美的自包含单文件 HTML**（左侧导航、代码高亮、复制按钮、响应式），也可 `--format md` 退回 Markdown。

只填这 7 个固定章节，不增不减、不画图、不做架构分析。目标读者是"要调用这个方法的人"，关心的是：怎么调、传什么、返回什么、什么时候会抛异常。

## 何时使用

- 用户输入 `api-doc <方法>` 或"为 xxx 生成接口文档 / 方法接口文档 / 调用说明"
- 要把某个对外暴露的 Service / Facade / RPC / Controller 方法整理成给其他开发调用的契约
- 接口联调、对外提供 API、SDK/RPC 说明、需求交付时附带的接口清单

## 何时不要用（改用 mfd 或 md 版）

- 只要纯文本 Markdown、不需要 HTML 样式 → 用 `method-api-doc`（md 版，本 skill 的姊妹）
- 需要时序图 / 流程图 / 状态机 → 用 `mfd`
- 需要设计决策（ADR）、质量属性（NFR）、失败模式、影响半径 → 用 `mfd`
- 目标是理解内部实现、代码评审、排障预研 → 用 `mfd`

一句话：**对外讲"怎么调"用本 skill；对内讲"怎么实现/为什么这么设计"用 mfd。**

## 参数解析

```
api-doc <方法定位> [输出位置] [--mode replace|append] [--format html|md]
```

| 参数 | 必填 | 说明 |
|------|------|------|
| 方法定位 | 是 | 方法名 `payBill` / 全限定 `com.x.Y#m` / 带签名 `com.x.Y#m(String,Long)` |
| 输出位置 | 否 | 目录或文件路径。未给时输出到对话中展示；用户要求落盘时放 `docs/` 下最相关子目录，不在 git 仓库则放当前目录 |
| --mode | 否 | `replace`（默认，覆盖）/ `append`（追加，前置 `## 修订 vN`） |
| --format | 否 | `html`（默认，自包含单文件 HTML，见步骤 4）/ `md`（Markdown，见步骤 3） |

## 执行步骤（严格顺序）

### 步骤 1：定位方法，重载消歧

1. 用 Grep/Read 找到目标方法源码。
2. 若同名方法有多个签名，按用户给的签名匹配；未给签名且存在多个时，**停下来让用户选**，不要猜。

### 步骤 2：采集模板所需信息（只采这些，不发散）

| 采集项 | 怎么采 | 填到哪一章 |
|--------|--------|-----------|
| 方法用途、适用业务场景 | 读方法体 + 类注释 + 调用方上下文 | 1. 方法说明 |
| 完整方法签名 | 方法声明原文（返回类型、方法名、参数列表） | 2. 方法定义 |
| 每个入参：类型、是否必填、含义、约束 | 形参 + 方法体内的非空/合法性校验 + 参数注解（`@NotNull` 等） | 3. 入参说明 |
| 返回对象的字段 | 返回类型的字段定义；若为统一包装（如 `data` 内对象）展开到调用方关心的层级 | 4. 返回值说明 |
| 一段最小可用的调用代码 | 依据签名构造，参照同包已有调用方写法 | 5. 调用示例 |
| 一份真实形态的返回 JSON | 依据返回结构造样例值 | 6. 返回示例 |
| 会抛的异常/错误码 + 触发条件 | 方法体内 `throw`、调用链已知异常、参数校验失败分支 | 7. 异常说明 |

**入参为 DTO / DTO 列表时的展开粒度**：只展开**方法体实际读取**的字段，不要把 DTO 全部字段铺开。在主入参表下另起一张子表（套用相同表头）列出这些字段，并加一句注明"该 DTO 还有其它字段，本方法仅使用以上字段"。

**返回基本类型 / 无统一壳时**：
- 第 4 章"字段名"列填 `(返回值)`，并注明"直接返回基本类型，无统一包装"。
- 第 6 章不要硬套 `{code,message,data}`，直接用真实返回值：基本类型就写裸值（如 `true`），对象就写该对象的真实 JSON。

**同一返回值有多种语义时**（如 `false` 既表示"入参为空"又表示"校验拒绝"）：在返回值说明的"说明"列**分语义列清**每种取值对应的业务含义，不要含糊合并成一句。

**目标是 Web Controller（HTTP 接口）时的适配**：当方法带 `@RequestMapping/@GetMapping/@PostMapping` 等映射注解，它对外是 HTTP 接口而非 Java 方法，按下面调整（其余章节不变）：
- **第 2 章方法定义**：先给 `HTTP方法 完整路径`（类级 `@RequestMapping` 路径 + 方法级路径拼全）和 `Content-Type`，Java 签名作为补充列在下面。
- **第 5 章调用示例**：改用 **HTTP 请求示例**（请求行 + 头 + JSON body），不要写 Java 注入调用。
- **入参**：`@RequestBody` 的 DTO 字段（含父类，需读父类）填入参表；`@RequestParam/@PathVariable` 直接作为独立入参列出。
- **必填性**：以 DTO 的 `@NotNull/@NotBlank/@ParamCheck` + 方法/Service 内手动校验为准。注意方法**未加 `@Validated/@Valid`** 时 JSR303 注解不一定强制触发，真正兜底的是 Service 里的手动校验——以实际校验逻辑为准，不要只看注解。

**信息缺失时的处理**：拿不到的字段不要编。必填性无法确定的入参，"必填"列填 `?` 并在"说明"里标注"待确认"；示例值不确定填 `xxx`。宁可标注待确认，不可臆造约束。

### 步骤 3：把内容组织成固定 7 章节（章节、表头、顺序都不可改）

先按下面的 Markdown 骨架组织 7 章节内容——它既是 `--format md` 的最终产物，也是默认套 HTML 模板前的**内容蓝本**。严格套用，标题用方法名（可中文业务名 + 英文方法名）。

````markdown
# 方法名称

## 1. 方法说明

说明这个方法是做什么的，适合在什么业务场景下使用。

## 2. 方法定义

```java
返回类型 方法名(参数类型 参数名, 参数类型 参数名);
```

## 3. 入参说明

| 参数名 | 类型 | 必填 | 示例值 | 说明 |
|--------|------|------|--------|------|
| 参数名 | String | 是 | xxx | 参数含义、限制条件 |

## 4. 返回值说明

| 字段名 | 类型 | 示例值 | 说明 |
|--------|------|--------|------|
| 字段名 | String | xxx | 返回字段含义 |

## 5. 调用示例

```java
示例代码
```

## 6. 返回示例

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

## 7. 异常说明

| 异常/错误码 | 触发场景 | 处理建议 |
|-------------|----------|----------|
| IllegalArgumentException | 参数为空 | 检查入参 |
| BusinessException | 业务校验不通过 | 根据错误信息处理 |
````

### 步骤 4：套用 HTML 模板输出（默认格式）

**本 skill 默认产出 HTML**。把步骤 3 组织好的 7 章节内容套进本 skill 目录下的 `html-template.html`：
1. `Read` 同目录的 `html-template.html`，它含完整内联 CSS 与 7 章节骨架、`{{占位符}}` 和填充注释。
2. 把每章内容填进对应骨架：表格用 `<tr><td>` 行；入参"必填=是"用 `class="req"`、"否"用 `class="opt"`；代码/JSON 块放 `<pre>`，可用 `.k/.s/.c/.n` span 做关键字/字符串/注释/类型高亮；md 里的 `>` 引用块转成 `<blockquote>`（模板已含其样式）。
3. **HTTP vs RPC 二选一**：模板的 meta 卡片和第 2 章各给了 (A)HTTP / (B)RPC 两版（一版是 HTML 注释）。HTTP Controller 用 (A) `tag-http`+路径；RPC/Service 用 (B) `tag-rpc`+`类#方法`标识，并**删除用不到的那版**，第 2、5 章按 Java 形式（签名 / Java 调用）填。
4. **图标保持原样**：左侧导航与「方法说明」的图标是模板内置的内联 SVG（Lucide 线性风格，`stroke="currentColor"` 自动继承分色/选中变白），属于固定 UI 资产——整体复用、**不要替换成 Unicode 字符或 emoji**（字形尺寸/基线不一会导致图标与文字错位）。7 章图标依次为 file-text / code / list / braces / play / corner-down-left / alert-triangle；增删章节时新图标沿用同款线性 SVG。
5. **"模板只读"指不回写 `html-template.html` 本身**；输出文件中按上面规则删改 HTTP/RPC 块、增减表格行都是正常的。

**落盘**（仅当用户指定了输出位置）：
- 落盘前 `ls` 目标目录，若已有同类接口文档，先 Read 一份，保持表头/风格一致。
- 用 heredoc（`cat << 'EOF' > xxx.html`）写出，**禁止用 Write**（含大量 `<` `>` 与样式，且通常接近/超过 500 行阈值）；输出文件名同方法名（扩展名 `.html`）。
- 未指定输出位置时，在对话中告知并给出完整 HTML 内容或建议落盘路径。

**`--format md` 时**：跳过 HTML 模板，直接交付步骤 3 的 Markdown。落盘前 `ls` 目标目录对齐已有文档风格；内容预估超过 500 行用 `cat << 'EOF' > <file>` heredoc 分段写入，禁止 Write。

## 质量要求

- **7 个章节一个不少、一个不多**，顺序固定，表头字段不可改。
- 入参/返回/异常表**每个字段一行**，不要把多个字段挤一格。
- 调用示例与返回示例必须**前后一致**（示例里传的参数能对上返回的 data）。
- `code/message/data` 为统一返回壳；若该方法直接返回业务对象而非统一壳，第 6 章用真实返回结构，并在第 4 章注明"直接返回，无统一包装"。
- 全程中文描述；遵循项目约定（如 `CtmException` 中文消息）。
- **HTML 输出必须自包含、可独立打开**：CSS/JS 全内联、不引外部资源，图标用模板内置 SVG。

## 常见错误

| 错误 | 纠正 |
|------|------|
| 顺手画了时序图/流程图 | 本 skill 不画图，要图用 mfd |
| 漏掉第 7 章异常说明 | 7 章必须齐全，无异常也要写"无显式抛出异常" |
| 入参"必填"列全填"是"未核实 | 依据方法体校验判断；不确定填 `?` 标"待确认" |
| 返回示例 JSON 与返回值说明表对不上 | 两者必须同源，字段一一对应 |
| 把内部私有方法实现细节写进文档 | 只写调用方关心的契约，不暴露实现 |
| 用 Write 直接写 HTML | 含大量 `<`、`>` 与样式，必须用 heredoc 分段写，否则易转义/截断 |
| 把模板内置 SVG 图标换成 emoji/字符 | 图标是固定 UI 资产，整体复用、勿替换（字形尺寸/基线不一会错位） |
